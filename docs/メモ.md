## Структура папок

1. `cmd/api/main.go` - точка входа;
2. `sql` - хранит SQL-скрипты;
3. `internal/` - внутренняя логика приложения. код из неё НЕ может быть импортирован в другие аппки;
4. `internal/handlers/` - хранит логику обработки каждого API-запроса;
5. `internal/models/{task.go}` - хранит структуры данных, которыми будем оперировать.
6. `internal/database/database.go` - хранит методы для работы с БД (пока только для коннекта с БД);

## [docker-compose.yml](../docker-compose.yml)

### Директива `services/{postgres}/image`

Директива `image` указывает, какой готовый Docker-образ нужно скачать из реестра (Docker Hub) и запустить для конкретного сервиса. Это альтернатива директиве `build`, которая собирает образ локально из `Dockerfile`.

Если указать и `image`, и `build` одновременно, `docker-compose` соберет образ из `Dockerfile`, но назовет его так, как указано в `image`.

`-alpine`:

- быстрое скачивание;
- безопасность выше обычной;
- маленький размер (десятки МБ);
- берем для готовых сервисов (Postgres, Redis, Nginx, RabbitMQ).

### Директива `volumes`

секция `volumes` (тома) нужна для того, чтобы связать файловую систему внутри контейнера с файловой системой реального компьютера (хоста).

>**ВАЖНО!** Необходимо в конце зарегистрировать именованные контейнеры:
>
>```yaml
># наверху всё остальное
>volumes:
>    db_data:
>```
>

## PostgreSQL

### [UC-001] Менеджмент БД через терминал + `psql`:

```bash
# стоп контейнеров и снос volumes
docker compose down -v

# запуск контейнеров
docker compose up -d

# запуск утилиты psql, для менеджмента бд
# `-it`: интерактивно
docker exec -it <название_сервиса_постгри> psql -U <юзермище_юзера_постгри> -d <величаво_постгри>

# помацать сущность
\d <сущность>

# можно любую sql'лятину писать, например:
select * from tasks;

# свалить
\q
```

### [UC-002] Не работает коннект

Поменять порт, например на `"5433:5432"`.
